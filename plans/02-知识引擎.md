## 1. 设计理念

### 1.1 从"作者本位"到"知识本位"

传统语雀文档按作者、知识库组织。NovaBot 的知识引擎按**知识主题**组织。

```
传统：
语雀知识库 → 作者的文档列表

NovaBot：
知识主题 → 聚合的文档 + 去重的核心内容 + 个人思考旁注
```

### 1.2 知识卡片

**核心概念**：将相似文档聚合为一张"知识卡片"

```
┌─────────────────────────────────────────────────────────────┐
│                    知识卡片：爬虫入门                         │
```

## 2. 功能设计

### 2.1 文档检索

**输入**：关键词或自然语言问题
**输出**：相关文档列表 + 摘要

**MVP 实现**：

```python
def search_docs(query: str) -> list:
    """
    1. grep 搜索关键词
    2. 按匹配度排序
    3. 返回文档列表 + frontmatter 元数据
    """
    results = []
    for md_file in yuque_docs.glob("**/*.md"):
        if query.lower() in md_file.read_text().lower():
            fm = parse_frontmatter(md_file)
            results.append({
                "id": fm["id"],
                "title": fm["title"],
                "author": fm["author"],
                "book_name": fm["book_name"],
                "path": str(md_file),
                "summary": extract_summary(md_file, query)
            })
    return ranked(results)
```

**后续升级**：向量检索（RAG）

### 2.3 知识卡片生成（Phase 2）

**触发**：首次检索某主题时自动生成

**流程**：

```
1. 检索相关文档（向量相似度 > 阈值）
2. 提取核心知识点（LLM）
3. 去重合并
4. 提取个人思考（从文档正文识别）
5. 存储为 knowledge_cards/{topic}.json
```

**增量更新**：

```
当文档变更时：
1. 找到关联的知识卡片
2. 只重新生成受影响的部分
3. 保留用户的 personal_notes
```

## 4. 检索策略

### 4.1 检索方案选择

**核心决策**：复用 gno 作为检索引擎

| 方案 | 说明 | 推荐 |
|------|------|:----:|
| 直接调用 gno CLI | subprocess 调用 | ❌ 性能开销 |
| **通过 MCP 调用 gno** | gno mcp serve → AstrBot MCP | ✅ 推荐 |
| 自己实现检索 | grep + 向量 | ❌ 重复造轮子 |

**gno MCP 接入**：

```json
// AstrBot MCP 配置
{
  "command": "/home/admin/.bun/bin/gno",
  "args": ["mcp", "serve", "--index", "yuque"]
}
```

**gno 已索引语雀文档**：
- 索引路径：`/home/admin/yuque-docs`
- 支持命令：search（BM25）、query（混合）、ask（AI 回答）

### 4.2 MVP：通过 MCP 调用 gno

```python
# NovaBot 不直接实现检索，而是调用 gno MCP 工具
# gno MCP 提供的工具：
# - gno.search: BM25 关键词搜索
# - gno.query: 混合/语义搜索
# - gno.ask: AI 生成答案（带引用）

# 在 Plugin 中，LLM 会自动调用这些工具
# 无需手动调用，只需确保 gno MCP 已配置
```

### 4.3 gno 搜索模式对比

| 模式 | 命令 | 速度 | 适用场景 |
|------|------|------|----------|
| 关键词 | `gno search` | 即时 | 精确匹配 |
| 向量 | `gno vsearch` | ~0.5s | 相似语义 |
| 混合 | `gno query` | ~2-3s | 平衡召回 |
| AI 回答 | `gno ask --answer` | ~3-5s | 带引用的回答 |

### 4.4 检索流程（MVP）

```
用户提问
    ↓
AstrBot LLM 判断需要检索
    ↓
调用 gno.query MCP 工具
    ↓
gno 返回相关文档列表
    ↓
LLM 基于文档生成回答
    ↓
添加来源标注 + 延伸问题
```

### 4.5 后续优化（Phase 2）

- **知识卡片生成**：基于 gno 的索引数据，用 LLM 聚合相似文档
- **增量更新**：yuque2git 同步后，调用 `gno update` + `gno embed`

## 5. 智能推送

> 解决痛点：知识库管理成本高

### 5.1 问题

当前 yuque2git 推送：
- 点击发布就推送 → 噪音大
- 不知道改了什么 → 信息密度低
- 无法判断价值 → 用户疲劳

### 5.2 解决方案

**LLM 判断变更价值**：

```
文档变更 → 提取 diff → LLM 分析 → should_push?
                                    ├── true → 生成摘要 → 推送
                                    └── false → 记录日志，不推送
```

### 5.3 判断标准

| 类型 | 推送 | 示例 |
|------|:----:|------|
| 实质性内容变更 | ✅ | 新增章节、重写段落、更新结论 |
| 知识增量 | ✅ | 新增代码示例、补充参考资料 |
| 格式调整 | ❌ | 换行、缩进、标题格式 |
| typo 修正 | ❌ | 错别字、标点符号 |
| 元数据更新 | ❌ | frontmatter 字段变更 |

### 5.4 Prompt 模板

```
分析以下文档变更，判断是否值得推送给社团成员。

文档标题：{title}
作者：{author}
变更内容：
{diff}

判断标准：
1. 实质性变更 vs 格式/typo 调整
2. 是否有新的知识增量
3. 是否对他人有价值

返回 JSON：
{
  "should_push": true/false,
  "reason": "判断理由",
  "summary": "变更摘要（如果值得推送，不超过100字）",
  "change_type": "content/format/typo/metadata"
}
```

### 5.5 推送格式

```
📄 《{doc_title}》有更新

{summary}

✍️ {author} · {update_time}
📎 查看原文：{doc_url}
```
